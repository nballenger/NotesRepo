JavaScript is a language almost entirely associated with the web. It is executed by one of a number of interpreters built into a web browser (though it can execute in certain other contexts as well). It's an extremely flexible language, and despite being deeply object-oriented can support functional and imperative programming styles. There are a number of features that make JavaScript interesting to work with—a short list would include prototypical inheritance, the sandboxed/host object execution framework, and the growing number of extremely powerful libraries (jQuery, AngularJS, NodeJS, etc.)

JavaScript has a lot of oddities and hidden gems. A few in brief:

  • The full language spec is a bit of a hodge-podge of features. To be effective, many developers find that working with a constrained subset of the language produces more reliable, portable code. See Douglas Crockford's Javascript: The Good Parts for a full treatment of that idea.
  • Unlike programs in many other interpreted languages, JavaScript code is likely to be read and executed by a number of different interpreters depending on the client it is called with. Each of the major browsers maintains separate JavaScript engines, and writing code that works predictably in all of them can be challenging. While projects like jQuery can bridge many of the differences between engines, it's helpful to have sites like Can I use... that keep track of which engines support which API features.
  • JavaScript is the only language I know that has no native Input/Output facilities. Since all execution is carried out within a "sandbox," JavaScript can only do I/O through a "host object" in the interpreter that dictates what the language can see and effect.
  • Though there is no native concept of access control to object members, a common practice has been built around using closures to create ersatz private and protected methods and properties. It's reasonable at this point to say that you can only understand complex, modern JavaScript if you are extremely conversant with how it handles closures. See chapter 5 in John Resig and Bear Bibeault's book Secrets of the JavaScript Ninja for a good explanation of them.
  • Though not a part of the language itself, understanding the API of the Document Object Model (the most common of the possible host objects you might encounter) is crucial to understanding client-side JavaScript. They're not as tightly bound as something like XML and XSLT, but since the DOM is the major way of performing I/O of any kind, there's relatively little that doesn't involve it at some level.
  • Driven by how uneven the built-in language features can be, and the number of different interpreters code might be called by, there are successful projects to create things like CoffeeScript, a more rigid meta-language that compiles to very portable, reliable JavaScript source.

Given all that, you'd be forgiven for thinking JavaScript is more trouble than its worth. However, once you get over the initial complexity of finding the right tooling and feature subset, there's actually a very expressive language in there. Plus, what else are you going to use for client side web stuff, Cold Fusion?

Datatypes

Summary of Type System

There are two major kinds of data in JavaScript: objects and primitives. All objects are instances of other 'constructor' objects, in a prototypical hierarchy tracing back to the Object constructor. Primitives are one of five types: undefined, null, boolean, string, and number.

Idiosyncracies

The boolean, string, and number primitives each have a corresponding constructor: Boolean, String, and Number. Those constructors can be used to create objects with various appropriate methods, so you can do things like get the length of a String. It can in fact be difficult to spot the existence of the primitives, because JavaScript will automatically coerce a primitive into an object of the appropriate type if a method is called on it.

A note on coercion of primitives:

JavaScript achieves dynamic typing through type coercion. That's distinct from type conversion, in that a coerced value is typically only shifted to another type temporarily while some operation (comparison, method call, etc.) is performed. While this is designed to operate behind the scenes to make coding easier, it is worthwhile to know if and when coercion will occur.

As Mark Daggett says in Expert JavaScript:

In JavaScript, coercion is always about simplifying complex objects to a primitive form or converting between two primitive types. You cannot coerce a number into an array, but you can coerce an array into a number.

Here are the basic rules of primitive coercion in JavaScript:

  • If you attempt to concatenate a string with a non-string object or primitive JavaScript will:
    1. If the non-string is a primitive of another type, coerce to that primitive's object type.
    2. Call the toString() method on the non-string object. If a string primitive is returned, use that in the concatenation operation.
    3. If toString() does not return a string primitive, call the object's valueOf() function. If valueOf() does not return a string primitive, a TypeError exception is thrown.
  • If you attempt an arithmetic operation on a non-number, JavaScript will:
    1. If the non-number is a primitive of another type, coerce to that primitive's object type.
    2. Call the valueOf() method on the non-string object. If that returns a number primitive, use that in the operation.
    3. If valueOf() did not return a number primitive, call the object's toString() method. If that returns a string primitive, coerce that to a String object and call that object's valueOf() method.
    4. If no number primitive is found, throw a TypeError exception.
  • When attempting an equality operation between two values, JavaScript will:
    1. Attempt to coerce both operands to primitive types.
    2. Compare the primitive values.

Simple Types

In JavaScript, the 'simple' datatypes (roughly mapping to 'scalar' here) are instances of the primitives (boolean, string, number) and their attendant constructor objects (Boolean, String, Number).

Numbers

All JavaScript numbers, whether primitive or instances of Number, are stored internally as 64-bit floating point. The following are examples of numeric primitives/literals:

100        // The integer 100 
0.01       // The decimal fraction 1/100
1e2        // The exponent form of the integer 100
1e+2       // The exponent form of the integer 100, with optional unary positive on exponent
-100       // The negative integer -100
1e-2       // The exponent form of the fraction 1/100, using unary negation on exponent
NaN        // Literal value indicating "Not a Number", result of 0/0
Infinity   // Literal value indicating a number larger than 1.79769313486231570e+308
-Infinity  // Literal value indicating a number smaller than -1.79769313486231570e+308

Technically the last three aren't primitives, they're static properties of the Number class, but they can be treated as primitives in equality operations.

Characters

JavaScript does not have a separate primitive or object representing single characters. However, it is possible to isolate characters within a String object using methods like the following.

// Methods on a string primitive coerced momentarily into a String object:
"abc".charAt(0);        // Returns "a"
"abc".charCodeAt(0);    // Returns 97

// Methods on a persistent String object:
var alpha = new String("abc");
alpha.charAt(0);        // Returns "a"
alpha.charCodeAt(0);    // Returns 97

It is also reasonable to use single character string primitives and String objects, or to treat String objects as arrays of string primitives each one character in length:

"a"                       // Single character string primitive
var a = new String("a");  // Single character String object
var b = new String("bcd");
b[0]                      // String primitive "b" from object b
b.charAt(0);              // String primitive "b" from object b

Strings

JavaScript string primitives are double or single quoted character sequences, which may contain any Unicode character except their enclosing quote mark, the backslash (escape) character, and control characters. Quotes, backslashes, and control characters are acceptable if prefaced with the backslash escape character. The following are examples of string primitives:

'abcdef'                          // Single quoted string
"abcdef"                          // Double quoted string
'can\'t'                          // Single quoted string with escaped single quote
"He said, \"Oh, my.\""            // Double quoted string with escaped double quotes
"\u0041"                          // Code point escape indicating character "A"

Naming Conventions

Names in JavaScript are made of alphanumeric characters plus underscores.

JavaScript names are used for: statements, variables, parameters, properties, operators, and labels.

Additional conventions (cribbed from Douglas Crockford's very useful Code Conventions for the JavaScript Programming Language):

  • JavaScript files use the extension .js.
  • The underscore should not be the first or last character of a name.
  • Most variables and functions should start with a lower case letter.
  • Constructors (anything that needs to be used with new) should have an upperc ase first letter.
  • Variables intentionally in the global scope should be in all caps.

Input

JavaScript has no native input/output capacity. All communication between a JavaScript program and its environment is mediated by a "host object." In many cases the host object represents a web browser client program, but in instances like NodeJS can represent a server with access to far more extensive system resources.

The existence of host objects goes a long way toward explaining how JavaScript execution can be so divergent in different browsers. It isn't simply that the program is compiled for hardware with different capabilities (think of the difference in datatype size in C depending on the memory model of the hardware)—it's that each engine's host object may present a subtly (or wildly) different means of performing the same I/O tasks.

From STDIN

Though it has no access to STDIN, there are numerous ways to provide interactive input to a JavaScript program. Within a web browser JavaScript can set event listeners on elements of the DOM in order to register keypresses, mouse movements, and state changes in input elements. A program which echoed keypresses to an onscreen element might look like the following:

<!DOCTYPE html>
<html>
<head>
  <title>JS Keystroke Echo</title>
  <style type="text/css">
  #display { border: 1px solid black; padding: 20px; min-height: 200px; }
  </style>
</head>
<body>
  <p>Above display area.</p>
  <p id="display"> </p>
  <p>Below display area.</p>

  <script type="application/javascript">
  var display = document.getElementById('display');
  function echoKeystroke(e) {
    display.innerHTML = display.innerHTML + String.fromCharCode(e.keyCode);
  }
  document.addEventListener("keyup", echoKeystroke, false);
  </script>

</body>
</html>

Output

To STDOUT

JavaScript has no access to STDOUT, but does have the ability to print messages and values to a JavaScript console within the web browser:

console.log("Got to this point, value of x is " + x.valueOf());

Note that support for this feature is purely browser dependent, though it is available natively in Chrome, Safari, Opera, IE9+, and Firefox with the addition of Firebug.

To Files

No filesystem access is allowed from the host object available in browsers. The host object in NodeJS can grant access to the underlying filesystem, but only on the server side. Check out http://nodejs.org/api/fs.html for more information.

JavaScript does however have access to persistence on the client side through HTML5's Web Storage capabilities. In browsers that support Web Storage, access is through a localStorage global object that will indefinitely persist key/value pairs as strings:

localStorage.setItem("myKey","myValue");
var a = localStorage.getItem("myKey");
