#!/usr/bin/env perl
use warnings;
use strict;

my $message = "Hello, world!";
print $message;

Save as hello-world.pl, then execute with perl ./hello-world.pl

Created in 1987 by Larry Wall for general purpose unix scripting. Did (and does) that job well, but it got picked up for processing data in web requests in the '90's when CGI scripting got popular. It's distributed in some form with most *nix systems, and while it shines at text processing and system administration, it shows up in lots of different applications. Occasionally accused of being a "write-only" language, it has a highly adaptable syntax that can become very idiomatic very quickly.

Bearing in mind that I like perl, it's fair to say it's got more idiosyncracies than almost any other language (barring purposely obfuscated ones like Malbolge). There are specific examples in the sections below, but here's a few of the things that make perl different:

  • You can directly manipulate the tables that hold all user-defined and built-in symbols('sigils' in perl-speak)—want to rename the built-ins? Go for it. Dynamically reworking your variable names at runtime? Sure, why not.
  • Perl tries very hard to do something with any sequence of sigils and operands you hand it. It's not just weakly typed, it actively (eagerly!) does type conversions as and when necessary to keep the flow of execution alive.
  • The default behavior is to eschew warnings and allow loose interpretation of code. You have to actively turn on more predictable behavior with the warnings and strict pragmas.
  • The perl motto is "There's more than one way to do it." That's no joke—every time you learn the "right" way to do something, there's a different "right" way waiting in the wings.
  • Don't like explicitly declaring nested data structures? You're in luck! Perl loves to infer them from the way you reference a variable, so my %x; $x{'a'}[0]{'b'} = 1; creates a hash inside a list inside a hash, automagically.
  • Implicit variables! If you see a function call with no obvious arguments, like chomp() or even just chomp, it's actually operating on the variable $_, or possibly @_, depending on context. Perl knows you hate typing (we're all busy people), so it'll just stash a value in there that it infers from context, like the current line of an input stream.

Truth Values

For boolean decisions, here is how perl derives true/false from values:

  • If the value is numeric, 0 is false—all non-zero numbers are true.
  • If the value is a string, '' (the empty string) is false, all non empty strings are true, except '0'.
  • If the value is another kind of scalar, convert it to a number or string and try again.
  • If the value is not a scalar (array or hash), convert it to a scalar and try again.

Assignment

Operator Description Example
= Set left operand to value of right operand. 
my $a = 10;
my $b = 15;
my $c = $a + $b;  # $c set to 25 (10+15)
+= Set left operand to its current value plus the value of the right operand. 
my $a = 10;
$a += 10;  # $a set to 20 (10+10)
-= Set left operand to its current value minus the value of the right operand. 
my $a = 10;
$a -= 5;  # $a set to 5 (10-5)
*= Set left operand to its current value times the value of the right operand. 
my $a = 10;
$a *= 3;  # $a set to 30 (10*3)
/= Set left operand to its current value divided by the value of the right operand. 
my $a = 10;
$a /= 5;  # $a set to 2 (10/5)
%= Set left operand to its current value modulo the value of the right operand. 
my $a = 10;
$a %= 3;  # $a set to 1 (10%3)
**= Set left operand to its current value to the power of the right operand. 
my $a = 10;
$a **= 2;  # $a set to 100 (102)
.= Set left operand to its current value concatenated with the value of the right operand. 
my $a = 'abc';
$a .= 'd'  # $a is set to 'abcd' ('abc'.'d')
